# 基本設計書 (Basic Specification) - V1.0

## 1. システム全体俯瞰

本システムは、ユーザーが撮影した映画チケット（感熱紙）の画像から、Google Cloud Vision APIを用いて文字情報を抽出し、TMDb API経由で映画メタデータを紐付け、Supabase（PostgreSQL）に保存するPWA（Progressive Web App）である。

### 1.1 画面一覧

1. **ダッシュボード（映画棚）画面：** 過去に登録したデジタル半券が美しく並ぶメイン画面。
2. **チケットスキャン（カメラ）画面：** チケットを撮影・アップロードする画面。
3. **データ補完・編集（下書き）画面：** OCRがサジェストしたテキストをユーザーが微調整・確認する画面。
4. **半券詳細（エモUI）画面：** 生成されたリッチなデジタル半券の表示、およびSNSシェア画面。
5. **座席お好み分析（マイページ）画面：** 蓄積された座席データから傾向を可視化する画面。

## 2. 画面遷移とUI/UX設計 (Screen Transition & UX)

### 2.1 コアユーザーフロー（目標：5秒で記録完了）

```
[1. 映画棚 (ダッシュボード)] 
       ▼ 「＋」ボタンをタップ
[2. チケットスキャン画面]
       ▼ カメラ起動 ── チケットを撮影・送信
       ▼ (裏側で Cloud Vision API ─> TMDb API が走る：ローディングアニメーション)
[3. データ補完・編集画面] ※OCR結果が自動入力された状態
       ▼ ユーザーによる微調整（評価星タップ、ひと言メモ入力）
       ▼ 「アーカイブを生成する」ボタンをタップ
[4. 半券詳細画面] ※エモいデジタルチケットの生成
       ▼ コレクションへの追加完了 ＆ SNSへワンタップシェア

```

### 2.2 画面別要件定義

#### ① ダッシュボード（映画棚）

* **機能要件：**
* ユーザーがこれまでに登録した `movie_tickets` を `show_timestamp`（上映日時）の降順でグリッド表示（2列〜3列）。
* 表示要素は、TMDbから取得した「ポスター画像」をベースに、上部に「劇場名」「鑑賞日時」をチケットの半券風にクリッピングしたグラフィックUI。
* 画面右下に固定（Fixed）で、チケットスキャン画面へ遷移するフローティングアクションボタン（FAB）「＋」を配置。



#### ② データ補完・編集（サジェストUI）

* **UX要件（ライト層離脱防止のための超重要画面）：**
* OCRの解析が完了すると、システム側で推測した項目（作品名、劇場名、上映日時、座席番号）が**最初から入力ボックスに「下書き」として入った状態**で遷移する。
* **作品名サジェスト：** TMDbから検索された候補（最大3件）をカード形式で表示し、万が一誤認識していた場合はタップで正しい作品に切り替え可能とする。
* **座席番号：** OCRが検知した「J-12」などのテキストを表示。もし空欄だった場合、または間違っている場合のみ、ユーザーが「列（アルファベット）」「番（数字）」のドラムロール（ピッカー）またはテキストで簡易修正できるUI。
* **手入力ステップの最小化：** ユーザーが「1文字もタイピングせず、タップだけで完了できる」状態を理想とする。

## 3. 機能仕様とバックエンドロジック (Functional Logic)

### 3.1 OCRパース＆サジェストアルゴリズム（Next.js Route Handlers内処理）

Google Cloud Vision API から返却される `textAnnotations`（構造化テキスト）のパース処理は、完全一致ではなく以下の「確率ベースの正規表現ルール」を適用してサジェストフォームへ流し込む。

1. **上映日時の特定：**
* テキストブロックから `(19|20)\d{2}[/\-.](0[1-9]|1[0-2])[/\-.](0[1-9]|[12]\d|3[01])`（YYYY/MM/DD）および `\d{1,2}:\d{2}`（hh:mm）のパターンを抽出。
* 当日または前日の日付を最優先候補としてタイムスタンプ（`show_timestamp`）に変換する。


2. **座席番号の特定：**
* アルファベット＋数字の組み合わせ（例：`([A-Z])\s?[-−‐－]?\s?([0-9]{1,2})`）を検出。
* マッチした文字列を `seat_raw` に格納。同時に、第1グループを `seat_row`（列）、第2グループを `seat_number`（番）に分解して変数に保持する。


3. **作品名の特定：**
* 劇場名や日時、金額（「￥2,000」「一般」等）の既知の文字列ブロックを除外。
* 残った最長文字列（または特定ブロックに挟まれた文字列）を「作品名候補」として抽出し、そのままTMDb APIの `search/movie` エンドポイントへリクエストを投げる。

## 4. 将来拡張：座席お好み分析ロジック (Seat Analytics Logic)

チームからの「座席の好みを分析・レコメンドしたら面白そう」というフィードバックを具現化するため、マイページ内に以下の分析セクションを実装する仕様を定義。

### 4.1 傾向分析クエリ（PostgreSQL / Supabaseでの集計イメージ）

データベースの `movie_tickets` テーブルに保存された `seat_row`（列）と `seat_number`（番）をベースに分析を行う。

* **「お気に入りゾーン」の算出：**
* `seat_row`（列）の頻出度（COUNT）を集計し、ユーザーが「前方（A〜E列）」「中央（F〜J列）」「後方（K列以降）」のどこに座る傾向があるかをパーセンテージで可視化する。
* `seat_number`（番）の偶数・奇数、あるいは数値のレンジから「左端」「中央」「右端」の偏りを算出する。



### 4.2 レコメンドロジック

ユーザーが新しいチケットを登録、またはマイページを閲覧した際、以下のスコアリングに基づく文言を表示する。

* **出力例：**
> 「佐藤さん、あなたは**『劇場の中央・やや後方（例：J列 10〜15番周辺）』**を好む偏愛傾向があります！このエリアはスクリーンの視界が最も均等になるUIデザイナーに人気の座席です。次に〇〇シネマズに行くなら、**J列の12番**が最もおすすめです。」

## 5. データモデル・APIマッピング (Data Mapping)

### 5.1 SupabaseへのINSERT処理マッピング

データ補完・編集画面でユーザーが「アーカイブを生成する」を押却した際、フロントエンドからバックエンドAPI（`/api/tickets/create`）を経由してSupabaseへ書き込まれるデータのマッピング表。

| UI入力項目 | 処理内容 / ロジック | 保存先カラム (`movie_tickets`) |
| --- | --- | --- |
| **確定した映画タイトル** | TMDb APIの検索結果からユーザーが選択した正式名 | `movie_title` |
| **ポスターURL** | TMDbから返却された `poster_path` の文字列 | `poster_path` |
| **確定した劇場名** | OCRサジェスト、またはユーザーの手動選択値 | `theater_name` |
| **確定した上映日時** | OCRサジェスト、またはユーザーの手動選択値（ISO形式） | `show_timestamp` |
| **座席番号（全体）** | OCRサジェスト、またはユーザー入力（例: "J-12"） | `seat_raw` |
| **座席（列）** | `seat_raw` からアルファベット部分を分離して抽出 | `seat_row` |
| **座席（番）** | `seat_raw` から数字部分を分離して整数（Int）に変換 | `seat_number` |
| **ユーザー星評価** | UI上の星タップUI（0.5〜5.0） | `rating` |
| **ひ言メモ** | テキストエリアへの自由入力値（任意） | `memo` |
| **OCR生のRAWデータ** | 精度向上のためCloud Vision APIの応答テキストをそのまま保存 | `raw_ocr_text` |
| **ユーザーID** | `supabase.auth.user().id` から自動取得 | `user_id` |

## 6. 開発フェーズとMVPスコープの再定義

基本設計の範囲を明確にし、開発のバースト（肥大化）を防ぐ。

* **PHASE 1 (MVP) - 本設計書の対象範囲：**
* 物理チケットのカメラ撮影 ──> Cloud Vision APIによるテキスト抽出。
* 特定3社（TOHO、イオン、109）のレシートパターンに対する「上映日時・座席・作品名」の簡易パースとフォーム自動サジェスト。
* TMDb連携による、デジタル半券（基本デザイン1種のみ）の自動生成と映画棚への保存。


* **PHASE 2 (GROWTH) - バックログ：**
* 「お好み座席分析・劇場別レコメンド」画面の実装（データはPHASE 1から溜め始める）。
* 予約完了メール（テキストコピペ）による補完機能の追加。
* デジタルチケットのデザインテンプレートの複数追加、および「チケットちぎり（分配）」によるバイラルマーケティング機能の実装。
