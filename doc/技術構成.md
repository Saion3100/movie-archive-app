# 技術構成・データ設計書 (Architecture & Database Design)

## 1. 技術スタック (Tech Stack)

| レイヤー | 選定技術 | 採用理由 |
| :--- | :--- | :--- |
| **Frontend / Backend API** | Next.js 15+ (App Router) | 画面とAPI Routeの一体開発。Anti-Gravity/Vercelでの高速デプロイ。 |
| **Language** | TypeScript | 劇場の複雑な文字列パースロジックを型安全に守るため。 |
| **Styling** | Tailwind CSS | チケットの「エモい」UI（半券のミシン目、グラデーション）を爆速で組むため。 |
| **Database / Auth** | Supabase | Postgresベース。行単位セキュリティ(RLS)により個人データを完全に保護。 |
| **External API** | TMDb API (The Movie Database) | グローバル標準の映画情報・高画質ポスターメタデータの取得。 |

## 3. データベース論理モデル (Supabase Schema)

### 3.1 `profiles` テーブル (ユーザープロファイル)
Supabase Auth の `users` テーブルと `id` (UUID) で1対1紐付け。

| カラム名 | 型 | 制約 | 説明 |
| :--- | :--- | :--- | :--- |
| `id` | `uuid` | PRIMARY KEY | Supabase Authの user_id |
| `updated_at` | `timestamp with time zone` | - | 最終更新日時 |
| `username` | `text` | - | 表示ユーザー名 |

### 3.2 `movie_tickets` テーブル (デジタル半券データ)

| カラム名 | 型 | 制約 | 説明 |
| :--- | :--- | :--- | :--- |
| `id` | `uuid` | PRIMARY KEY, DEFAULT: gen_random_uuid() | チケット一意ID |
| `user_id` | `uuid` | REFERENCES profiles(id), NOT NULL | 所有ユーザーID |
| `brand` | `text` | NOT NULL | 劇場ブランド ('KEISEI' \| 'TJOY' \| 'USCINEMA') |
| `theater_code` | `text` | NOT NULL | 4桁の劇場識別コード |
| `theater_name` | `text` | NOT NULL | 日本語の劇場名 (例: 'T・ジョイ蘇我') |
| `reserve_id` | `text` | NOT NULL | 予約番号・シリアル枝番 |
| `show_timestamp`| `timestamp with time zone`| NULLABLE | 上映日時 (T・ジョイは初期NULL可) |
| `movie_title` | `text` | NOT NULL | TMDbから確定させた正式な映画タイトル |
| `poster_path` | `text` | - | TMDbのポスター画像フルURLまたはパス |
| `rating` | `numeric` | DEFAULT: 0 | 5段階の星評価 (0.5刻み) |
| `memo` | `text` | - | ユーザーのひと言メモ |
| `raw_qr_string` | `text` | NOT NULL | 解析元のQRコード生データ (ログ用) |
| `created_at` | `timestamp with time zone`| DEFAULT: now() | 登録日時 |

## 4. セキュリティ方針 (Row Level Security)
SupabaseのRLSを有効化し、以下のポリシーを適用する。
- **Policy:** `auth.uid() == user_id`
- **対象操作:** `SELECT`, `INSERT`, `UPDATE`, `DELETE`
- **効果:** 自分以外のユーザーが登録したチケットデータには、API経由であっても絶対にアクセスできない状態を担保する
