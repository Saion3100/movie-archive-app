# 技術選定書：映画体験アーカイブWebアプリ（仮）

## 1. はじめに
### 1.1 本書の目的
本書は、「映画体験アーカイブWebアプリ」の開発における技術選定の結果と、その選定理由を記述したドキュメントである。要求分析において浮き彫りになった「入場用QRコードのデータ構造の限界」という制約を突破し、現実的な代替案である「物理チケットのOCRスキャン」および「半自動入力UX」を最小コスト・最大精度で実現するためのシステムアーキテクチャを定義する。

### 1.2 技術選定における最優先評価軸
1. **OCR（文字認識）の精度と堅牢性：** 劇場の暗いロビーや、スマホカメラの画質、シネコンごとの異なるレイアウトに対応できること。
2. **高速なプロトタイピング（スピード）：** MVP（最小限の製品）を素早く構築し、ユーザーテストに回せること。
3. **UI表現の自由度：** ペルソナ（佐藤航平・UIデザイナー）の所有欲を満たす「エモいUI（グラフィカルな半券）」を完全に再現できること。

---

## 2. システムアーキテクチャ概要
ユーザーがスマホカメラでチケットを撮影してから、デジタル半券が生成されるまでのデータフローと基本構成は以下の通り。

```
[User / PWA (Next.js)] 
       │ (1) チケット画像をアップロード
       ▼
[Backend (Next.js Route Handlers)]
       │ (2) 画像を送信
       ▼
[Google Cloud Vision API] (文字認識処理)
       │ (3) 構造化テキスト（RAWデータ）を返却
       ▼
[Backend (解析ロジック)] 
       │ (4) 正規表現等で「作品名」「日時」「座席」をパース
       │ (5) 作品名をもとに映画情報を検索
       ▼
[TMDb (The Movie Database) API] (作品情報・ポスター取得)
       │ (6) ポスター画像メタデータを返却
       ▼
[User / PWA (Next.js)] 
         (7) 自動補完された入力フォーム・プレビューを表示
```

## 3. コンポーネント別 技術選定

### 3.1 OCR（文字認識）エンジン
* **選定技術：`Google Cloud Vision API (DOCUMENT_TEXT_DETECTION)`**
* **比較対象：** `Tesseract.js` (クライアントサイドOCR), `AWS Rekognition`
* **選定理由：**
  * **圧倒的な日本語認識精度：** 感熱紙特有の薄い印字や、明朝・ゴシック等のフォント違い、劇場ロビーの低照度環境でも高い認識率を誇る。
  * **レイアウト解析能力：** `DOCUMENT_TEXT_DETECTION` モードを使用することで、レシート形式の縦並びテキストのブロック構造（座標情報）を保持したままパースできるため、「作品名」「上映日時」「座席番号」の位置関係を特定しやすい。
  * **不採用理由（Tesseract.js）：** 純フロントエンドで完結する魅力はあるが、スマートフォンのリソースを消費する点、および低画質・暗所での日本語認識精度が実用レベルに達しないためMVPのコア体験（爆速・手入力なし）を損なうと判断。

### 3.2 フロントエンド・フレームワーク
* **選定技術：`Next.js (App Router) + TypeScript`**
* **比較対象：** `Vite + React`, `Nuxt (Vue.js)`, `Flutter Web`
* **選定理由：**
  * **PWA (Progressive Web App) との親和性：** 独自ドメインでのネイティブアプリ風挙動（ホーム画面追加、スタンドアロン起動、高速なキャッシュ制御）を、プラグイン（`@ducanh2912/next-pwa` 等）で容易に実現可能。ストア審査を回避する戦略に合致する。
  * **サーバーサイド機能（Route Handlers）の一体化：** フロントエンドと同じリポジトリ・言語（TypeScript）で、Cloud Vision APIやTMDb APIを叩く安全なバックエンドAPI（BFF）を構築できるため、初期の検証スピードが最速となる。
  * **デザイナー要件の充足：** `Framer Motion` などのアニメーションライブラリとの組み合わせにより、半券が生成される際の演出など、デザイナー層に刺さるリッチなインタラクションを完全実装できる。

### 3.3 CSS / スタイリング
* **選定技術：`Tailwind CSS`**
* **比較対象：** `Vanilla Extract`, `Scss`
* **選定理由：**
  * ユーティリティファーストによる圧倒的な開発スピード。UIデザイナーからの微細なデザイン調整（ピクセルパーフェクトな余白、カラーコード指定）に対して、クラス名の追加だけで即座に応答できる柔軟性を評価。

### 3.4 外部映画データベース（メタデータ取得）
* **選定技術：`TMDb (The Movie Database) API`**
* **比較対象：** `Filmarks (API非公開)`, `OMDb API` (日本語データ不足)
* **選定理由：**
  * 無料で利用可能（商用利用は要申請）であり、日本国内で公開されるメジャー・インディーズ双方の映画情報・公式ポスター画像（高解像度）の網羅率が非常に高い。
  * OCRでパースした「作品名テキスト」をクエリとして曖昧検索（Search Movie）をかけることで、高確率で正しい映画IDおよびポスターURLを紐付けることができる。

### 3.5 インフラ / ホスティング
* **選定技術：`Vercel`**
* **比較対象：** `AWS (Amplify / ECS)`, `Render`, `Cloudflare Pages`
* **選定理由：**
  * Next.jsとの親和性が最も高く、CI/CD環境の構築が不要（GitHub連携のみでデプロイ完結）。
  * サーバーレス関数（Serverless Functions）の起動が高速であり、OCR処理のようなスパイクが発生しやすい（映画の終映直後にユーザーが一斉にアクセスする）スパイク耐性を低コストで担保できる。

## 4. 技術的課題（リスク）と対策（サーキットブレーカー）

### 4.1 OCRパースのシネコン別対応と「カオス化」リスク
* **リスク：** TOHO、イオン、109シネマズ等、シネコンごとにレシートの文字列パターン（「シネマ」「スクリーン」「座席」等）が異なるため、バックエンドの正規表現ロジックが複雑化・破綻する恐れがある。
* **対策：** 1. OCRロジック側で「完全な自動パース」をゴールとしない。
  2. Cloud Vision APIから返ってきたテキスト全体から「座席っぽい正規表現（例：`[A-Z]-[0-9]{1,2}`）」や「日時っぽい正規表現（例：`[0-9]{1,2}:[0-9]{2}`）」を確率ベースで抽出し、フォームに**「サジェスト（下書き）」**として流し込む。
  3. ユーザーには、間違っている部分だけを「タップして修正・選択」してもらうUI/UX（半自動補完）を提供し、システム側の完璧なパース義務をデザインで回避する。

### 4.2 将来的な「お好み座席分析」を見据えたデータ構造
* **リスク：** チームのフィードバックから、将来的に「座席の好み分析・レコメンド」へシフトすることが決定した。MVPの段階でデータ構造を縛りすぎると、将来のデータマイニングが困難になる。
* **対策：** * データベース（MVP時は暫定的に `Supabase` または `Prisma + PostgreSQL` を想定）の `Ticket` テーブルにおいて、座席番号を単なる文字列として保存するだけでなく、`theater_id`（劇場）、`screen_number`（スクリーン名）、`seat_row`（列: J列）、`seat_number`（番: 12番）のように、最小単位に分解（パース）してオプショナルで保存できるカラム設計をはじめから定義しておく。
