# 要求分析結果報告書：映画体験アーカイブWebアプリ（仮）

## 1. プロジェクトの背景と目的
### 1.1 背景
* **映画館のDXと情緒的ロス：** 劇場の入場システムが「厚紙の半券」から「感熱紙レシート」や「スマート入場（QRコード）」へ移行し、ペーパーレス化・低コスト化が進んでいる。
* **消費者のプレミアム化：** 映画観賞料金の値上げに伴い、映画館での鑑賞は単なるコンテンツ消費ではなく「特別な体験（推し活・ライフログ）」としての価値が高まっている。
* **市場の歪み：** 劇場側の「合理化・ペーパーレス化」の潮流と、ファンの「体験を一生モノとして手元に残したい」という「所有欲・情緒的ほし欲」の間に巨大なギャップ（歪み）が生じている。

### 1.2 目的
本プロジェクトは、入場後や数年で印字が消えてしまうペラペラの感熱紙レシートや画面上のQRコードを、一生色褪せない「資産（デジタル半券）」へと変換・コレクションするWebアプリケーション（PWA）の開発を目指す。他社アプリが「作品の評価・レビュー（映画そのもの）」にフォーカスするのに対し、本アプリは「その日、その劇場で、どう過ごしたか（自分自身と体験）」を美しくパッキングして保存することを主目的とする。

---

## 2. ターゲットユーザー像（ペルソナ）と利用シーン
### 2.1 ペルソナ：佐藤 航平（27歳・UIデザイナー）
* **行動特性：** 年に約60回劇場へ通うヘビーユーザー。
* **インサイト（悩み）：** 既存の映画管理アプリ（Filmarks等）は文字中心で味気なく、デザイナーとして納得できる美しいUIで「自分が映画館に行った証」を並べて眺めたい。
* **矛盾する欲求（ペイン）：** 映画に対する強い熱量（偏愛）を持つ一方で、「長文のレビューを書くこと」や「上映直後に細かなデータを手入力すること」は極めて面倒だと感じている（タイパの重視）。

### 2.2 利用シーン
* **シーン1：** 映画鑑賞直後、劇場の余韻が残るロビーや帰りの道中にて、極めて少ないステップ（目標5秒以内）で「映画体験」を記録する。
* **シーン2：** アプリ内の「映画棚」に美しく並んだデジタルチケットを眺め、所有欲を満たす。また、そのグラフィックをInstagramやX（旧Twitter）へワンタップでシェアし、フォロワーへ「スクショ映え」する形で自慢する。

## 3. 主要機能要件（初期構想）
1. **爆速記録（トリガー入力）：** 映画館を出た直後の5秒間で記録を完結させるための自動補完入力。
2. **デジタル半券の自動生成（エモいUI）：** TMDb等の外部映画データベースAPIや、ポスター画像と連携し、リッチなグラフィックによる「色褪せないデジタルチケット」を生成する。
3. **映画棚（コレクション）：** 生成された半券をグリッド状に並べ、視覚的にコレクションを楽しめるマイページ。
4. **座席コレクター機能：** チケットデータから「座席番号」を抽出し、劇場の白地図（座席配置図）を塗りつぶしていくゲーミフィケーション要素。

## 4. 技術的制約と課題（割り切りとピボットの経緯）
当初想定していた「入場用QRコードのカメラ読み取りによる一発データ抽出」について、調査の結果以下の致命的な障壁が判明した。
* **QRコードの仕様制限：** 各シネコン（TOHO、イオン、109等）で暗号化や独自データ構造が採用されており、外部アプリからは「作品の特定」までしかできないケースがほとんどで、日時や座席番号などの詳細データがパースできない。

このため、トリガー（入力方法）を以下の代替案へと切り替えて検討を行うこととなった。
* **代替案A：** 発券された物理チケット（感熱紙）をカメラで撮影し、OCR（文字認識）でデータを抽出する。
* **代替案B：** オンライン予約時に届く「予約完了メール」のテキストをパース（転送やコピペ、または自動連携）して抽出する。

## 5. チームからのフィードバック・指摘および要求の再定義
開発チーム内での議論により、上記代替案および主要機能に対して以下のリアルな指摘・懸念が挙がった。これらを要求のブラッシュアップおよび基本設計の前提条件として組み込む。

### 5.1 データ入力・トリガーに関する指摘
* **【指摘】メール連携の不確実性と手間の懸念**
    > 「メールのほうが正確に情報を保管できそうだが転送やコピペも面倒くさいし、メールも各社で内容が違うから同じ水準になるの？」
    * **要求への反映：** メールのパースは各シネコンごとの正規表現のメンテナンスコストが高い。また「転送やコピペ」という追加アクションはペルソナの「面倒くさい」を刺激するため、メイン導線とするには体験が劣る。
* **【指摘】スキャン（OCR）の汎用性の高さ**
    > 「映画館で発券したらメールは届かない。スキャンのほうが汎用性は高そう」
    * **要求への反映：** 現地発券派（シニア層やムビチケ利用、あるいは直前購入）を考慮すると、メールだけではカバーできない。物理チケットを撮影する「スキャン方式（OCR）」を第一優先の検討軸とする。
* **【指摘】スキャン（OCR）の技術的難易度**
    > 「スキャンもメールと同様各社フォーマットが違うから対応できるのか」
    * **要求への反映：** OCRでもフォントやレイアウト、劇場の暗いロビーでの撮影による認識精度のブレが予想される。基本設計では、Cloud Vision API等の検証（PoC）を進めると同時に、「完全に自動抽出できなくても、読み取れたものだけフォームに自動補完し、漏れた部分はユーザーがタップで微調整できる」ような、OCR＋半自動UIの設計を必須とする。
* **【指摘】ライト層獲得の壁**
    > 「できるだけ入力の手間を省かないとライト層では利用されない」
    * **要求への反映：** ペルソナ（ヘビー層）だけでなく、将来的なライト層拡大を狙うなら、手入力ステップの最小化がプロダクトの生死を分ける。作品名選択をレコメンドやGPS（現在地の最寄り劇場の上映スケジュール）から推測して補完するなどのUX工夫を基本設計に盛り込む。

### 5.2 座席機能に関する指摘とアイデアの転換
* **【指摘】座席コレクター機能の魅力不足**
    > 「座席コレクター機能はそこまでの魅力を感じない」
    * **要求への反映：** 劇場の白地図を自前で用意・マッピングする膨大な開発コストに対し、ユーザーが感じる価値（コレクション性）が限定的である可能性が示唆された。よって、MVPフェーズでの「劇場座席表の完全再現（グラフィカルな塗りつぶし）」は要件から除外（またはバックログへ降格）する。
* **【指摘】座席データのレコメンド・分析への昇華**
    > 「座席の好みがあるから分析して～がユーザの好み～劇場ならここがおすすめみたいなのあったら面白そう」
    * **要求への反映：** 単なる「塗りつぶし（スタンプラリー）」から、**「データ分析とレコメンド（価値の提供）」**へと機能をシフトする。過去の鑑賞履歴から「あなたは中央後方席を好む傾向があります。この劇場なら〇列目がおすすめ」といった、映画ライフを豊かにするパーソナライズ機能を将来的なコア価値として位置づける。
* **【指摘】業界データの限界**
    > 「座席配置配布してたらいいのにね」
    * **要求への反映：** 各劇場の座席配置データが公式にAPI等で配布されていないため、システム側で正確な座席配置を保持・再現するのは困難。やはり「テキストデータとしての座席番号（例：J-12）」をベースとした、簡易的かつ実用的な仕様（カード型コレクションや、簡易的な座席傾向分析）に着地させるのが現実的である。

## 6. 基本設計書に向けた方針（要約）
1. **MVPのトリガーは「チケットスキャン（OCR）＋半自動補完」を主軸に検証：** 技術検証（PoC）にて認識精度を測りつつ、UIでカバーする構成にする。
2. **「エモいUI」の維持：** 入力方法が泥臭くなっても、生成されるデジタルチケットのクオリティが高ければデザイナー層の所有欲は満たせる。生成エンジンの表現力にリソースを割く。
3. **座席機能の仕様変更：** 座席表のグラフィック再現は行わず、テキストデータ（座席番号）の蓄積による「お好み座席分析・レコメンド」へと要件をシフトし、基本設計のデータ構造を定義する。
